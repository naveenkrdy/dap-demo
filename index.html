DAP Selector Strategy

Element identification guide for digital adoption platform targeting

Recommended Selectors - Stable and Reliable

Semantic ID attributesStable
#report-title #expense-category #submit-btn #add-item-btn #proceed-btn
Every key field and CTA has a hand-authored, human-readable ID that describes its business purpose. These IDs do not change unless the business logic changes - which is exactly what you want for DAP tooltips and walkthroughs.
data-* custom attributesStable
[data-action="submit-expense"] [data-action="add-line-item"] [data-field="expense-category"] [data-screen="review"] [data-step="2"]
data-* attributes are the gold standard for DAP targeting. They are invisible to users, survive CSS refactors, and express intent directly. Every interactive element in this app has a data-action or data-field attribute for this reason.
ARIA label selectorsStable
[aria-label="Main navigation"] [aria-label="Expense line items"] [aria-label="Approval chain"] [aria-current="step"]
ARIA attributes exist for accessibility compliance - maintained rigorously by engineering teams. Removing them would break screen reader compliance. DAP can piggyback on this stability.
name attribute (forms)Stable
[name="expense_category"] [name="report_title"] [name="cost_center"]
Form name attributes are tied to the data model. Changing them breaks form submission and API payloads - making name attributes nearly immutable in production apps.

Selectors to Avoid - Brittle and Unstable

CSS class-based selectorsBrittle
.btn-primary /* breaks on refactor */ .card-head button /* positional */ .form-grid .field:nth-child(3)
CSS classes are for styling, not identity. A frontend rebrand that renames .btn-primary silently breaks all DAP steps targeting those classes with no warning.
Positional XPath / DOM pathBrittle
div > div:nth-child(2) > button /html/body/div[1]/main/section[1]/...
Absolute DOM paths break the moment any structural change occurs - adding a wrapper div, reordering sections, or inserting a feature-flag component.
Auto-generated IDsBrittle
#react-select-42__control #ember-view-1847 #mat-input-23
React, Ember, Angular generate numeric IDs at runtime. These numbers increment differently every page load. Targeting #react-select-42 works on one session and fails on the next.
Text-content matchingBrittle
button:contains("Submit for Approval") //button[text()="Submit for Approval"]
Text matching breaks on internationalisation, copy changes, and A/B test variants. If the product team renames the button, every DAP step targeting the old text silently breaks.

Edge Cases and Tradeoffs

Dynamic rows in the line-items tableContext-dependent
/* Row 1 - always present */ #line-amount-1, #line-desc-1 /* Row N - use data-attribute pattern */ [data-field="line-amount"][data-row="2"] [data-field="line-amount"] /* targets ALL amount fields */
Line items grow dynamically. Row 1 has a stable ID but rows 2, 3, N are generated at runtime. I addressed this by adding data-row and data-field attributes to every generated cell so a DAP can target specific rows or the entire column pattern. Tradeoff: a walkthrough saying "enter amount in row 1" assumes the user starts with one row - better to target the pattern and guide generically.
Multi-screen state - same DOM, different screensContext-dependent
/* Wrong: fires regardless of which screen is visible */ [data-action="proceed-review"] /* Better: scope to active screen */ [data-screen="entry"].active [data-action="proceed-review"]
This app uses a single-page pattern where all three screens exist in the DOM simultaneously. A DAP tooltip targeting a button ID will fire on any screen. The data-screen attribute on each section gives DAP a reliable hook to scope walkthrough steps to the correct screen.
Salesforce / Workday iframes in productionProduction consideration
/* Cross-origin iframes block DAP selector access entirely */ /* Same-origin iframes need explicit frame context */ dap_config: { frame: "#canvas-iframe", selector: "[data-field='amount']" }
In production Salesforce Lightning and Workday, many views render inside iframes. Cross-origin iframes are inaccessible to DAP overlays entirely due to browser same-origin policy. This is one of the most common reasons enterprise DAP deployments fail in the field - always confirm iframe architecture before writing the first walkthrough step.

Summary: Selector Priority Hierarchy

Priority 1 (Best): data-* custom attributes [data-action="submit-expense"] Priority 2: Semantic hand-authored IDs #submit-btn Priority 3: ARIA attributes [aria-label="Expense line items"] Priority 4: name attributes (forms) [name="expense_category"] ----------------------------------------------------------------------- Avoid: Auto-generated IDs #react-select-42 Avoid: CSS utility classes .btn-primary Avoid: Positional selectors div:nth-child(3) > button Avoid: Text content matching button:contains("Submit")
Core principle: target identity, not appearance. Anything that describes how an element looks or where it sits is fragile. Anything that describes what an element IS is durable. In enterprise apps where frontend teams ship weekly, this is the difference between a DAP walkthrough that works for 2 years and one that breaks in the next sprint.
